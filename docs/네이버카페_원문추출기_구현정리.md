# 네이버 카페 원문 추출기 구현 정리

## 1. 만들게 된 이유

네이버 카페에서 내가 작성한 게시글 목록을 수집하고, 각 게시글 본문을 나중에 NotebookLM 같은 도구에서 정리할 수 있도록 Markdown 파일로 저장하기 위해 만들었다.

초기에는 게시글을 바로 요약하려고 했지만, 로컬 휴리스틱 요약이 너무 허술했다. 그래서 방향을 바꿔서 확장 프로그램은 **원문 추출과 파일 저장**만 담당하게 했다.

한 번 더 검증한 기준은 이것이다.

- 요약이 그럴듯한가보다, 원문이 정확히 저장됐는가가 먼저다.
- 본문 검증 전 요약을 만들면 틀린 정보가 더 믿을 만해 보일 수 있다.
- 원문 Markdown 파일을 만든 뒤 NotebookLM에서 요약하는 편이 더 안전하다.

## 3. 파일별 역할

### manifest.json

Chrome 확장 프로그램 설정 파일이다.

- Manifest V3 사용
- 네이버 카페 페이지 접근 권한 설정
- `downloads`, `scripting`, `tabs` 권한 사용
- content script를 모든 frame에 주입

### postExtractor.js

게시글 목록 화면에서 `a.article` 요소를 찾는다.

추출하는 값:

- `title`
- `url`
- `articleId`
- `clubId`

같은 게시글이 여러 frame에서 중복으로 잡힐 수 있어서 `articleId` 기준으로 중복 제거한다.

### content.js

popup에서 보낸 수집 요청을 받아 현재 frame의 게시글 목록을 반환한다.

네이버 카페는 iframe 구조를 쓰는 경우가 많기 때문에 모든 frame에서 실행될 수 있게 했다.

### popup.js

확장 프로그램 popup의 중심 컨트롤러다.

담당하는 일:

- 버튼 이벤트 처리
- 현재 탭이 네이버 카페인지 확인
- 모든 frame에서 게시글 목록 수집
- 게시글 본문 추출 실행
- 결과 개수와 로그 표시
- export 버튼 활성화

### articleFetcher.js

게시글 본문을 가져오는 책임을 가진다.

시도 순서:

1. 네이버 카페 article API 호출
2. HTML 문서 fetch 후 본문 selector 탐색
3. 실패하면 popup.js에서 임시 탭 fallback 사용

API 응답이 `200`이어도 본문 payload가 비어 있는 경우가 있어서 `bodyLength`, `contentElementCount`, `rawTextCount` 등을 로그로 남기게 했다.

### rendered-tab fallback

API와 HTML fetch만으로 본문을 못 읽는 경우가 있었다.

그래서 최종 fallback으로 게시글을 비활성 임시 탭에서 실제 렌더링한 뒤, 모든 frame에서 DOM을 읽는 방식을 추가했다.

성공하면 로그에 아래처럼 표시된다.

```json
{"articleId":"62273","bodyLength":5230,"fallback":"rendered-tab"}
```

### exporter.js

추출된 결과를 파일로 저장한다.

지원 형식:

- JSON
- CSV
- 통합 Markdown
- 게시글 하나당 Markdown 파일 하나

현재 가장 권장하는 방식은 `Files` 버튼으로 글 하나당 `.md` 파일 하나를 저장하는 것이다.

### logger.js

실행 단계를 DevTools Console과 popup 로그에 남긴다.

로그를 남기는 이유:

- 어느 단계에서 실패했는지 확인하기 위해
- 본문 길이가 충분한지 검증하기 위해
- API 실패와 DOM selector 실패를 구분하기 위해

## 4. 실행 흐름

```text
사용자 클릭: Collect
        ↓
popup.js
        ↓
모든 frame에 postExtractor 실행
        ↓
게시글 목록 병합 및 중복 제거
        ↓
사용자 클릭: Document
        ↓
articleFetcher로 본문 추출 시도
        ↓
실패 시 rendered-tab fallback
        ↓
bodyText + metadata 저장
        ↓
사용자 클릭: Files
        ↓
게시글 하나당 Markdown 파일 저장
```

## 5. Markdown 저장 형식

각 게시글은 아래 형식으로 저장된다.

```md
---
title: 게시글 제목
url: https://cafe.naver.com/ArticleRead.nhn?clubid=...
articleId: 62273
clubId: 28969626
category: 게시판 이름
sourceType: rendered-tab
extractedAt: 2026-05-07T06:20:00.000Z
---

# Original Body

게시글 본문 원문...
```

## 6. 왜 요약 기능을 기본에서 뺐는가

처음에는 `category/topic/summary/keyConcepts/questionsToVerify` 형태로 문서화하려고 했다.

하지만 직접 확인해보니 자동 요약 결과가 다음 문제가 있었다.

- 본문 일부만 보고 요약할 수 있음
- 깨진 텍스트가 핵심 개념으로 잡힐 수 있음
- 요약이 그럴듯해서 원문 오류를 놓칠 수 있음

그래서 최종 설계는 다음처럼 바꿨다.

```text
확장 프로그램: 원문 추출 + 파일 저장
NotebookLM: 요약 + 질문 생성 + 학습 노트 정리
```

## 7. 검증 방법

개발 중 확인한 테스트:

```bash
node tests/sample-extractor-test.js
node tests/article-fetcher-test.js
node tests/article-documenter-test.js
```

사용자가 직접 확인해야 할 기준:

- `Collect` 결과 개수가 실제 목록 개수와 맞는가?
- `Document completed` 로그의 `bodyLength`가 너무 작지 않은가?
- `Files`로 저장된 `.md` 파일이 게시글 수만큼 생겼는가?
- `# Original Body`가 실제 게시글 본문과 일치하는가?

## 8. 현재 한계

- 네이버 카페 DOM 구조가 바뀌면 selector 수정이 필요할 수 있다.
- 비공개 글이나 권한이 없는 글은 추출되지 않을 수 있다.
- 이미지 자체를 파일로 다운로드하지는 않는다.
- 이미지 설명이나 본문 텍스트는 DOM에서 읽히는 범위까지만 저장된다.
- Chrome Web Store에 정식 등록한 상태는 아니므로 개발자 모드로 로드해야 한다.

## 9. 팀원이 보면 좋은 판단 기준

이 확장 프로그램은 완성된 지식 정리 도구가 아니라 **원문 수집 도구**다.

따라서 좋은 결과의 기준은 요약 품질이 아니라 아래에 있다.

- 원문이 빠지지 않고 저장됐는가?
- 게시글별로 파일이 분리됐는가?
- 나중에 NotebookLM에 넣어도 출처를 추적할 수 있는가?
- 실패했을 때 로그로 원인을 좁힐 수 있는가?
